# sportslivetrackerV1 — Implementation Plan

## What we're building

A real-time multi-sport telemetry dashboard. The problem it solves: live sports APIs have fresher data than your TV broadcast. This app pulls that live data, runs it through an AWS pipeline, adds AI color commentary, and lets you sync the UI to your exact broadcast delay via a slider.

**Reference project:** `/Users/rodrigo/code/recipe-ai-finder-v2-J` — same tech stack (Java 21 / Spring Boot / Next.js / Terraform / ECS Fargate / GitHub OIDC). We follow its structure and patterns wherever they make sense.

## How data flows (end-to-end)

```
API-Sports (external)
    ↓  (HTTP polling every 30s)
Producer Service (ECS Fargate)
    → normalizes raw API response into unified SportEvent model
    → publishes JSON to Kinesis Data Stream
    ↓
Lambda Function
    → reads from Kinesis
    → attaches precise UNIX timestamp
    → writes Event record to DynamoDB (events table)
    → calls Amazon Bedrock with sport-aware prompt
    → writes AI commentary to DynamoDB (summaries table)
    ↓
API Service (ECS Fargate)
    → polls DynamoDB every 5 seconds for new events
    → pushes events to connected frontend clients via SSE
    ↓
Frontend (Next.js, ECS Fargate)
    → receives live event stream
    → holds a client-side buffer of events
    → renders only events where (now - eventTimestamp) > broadcastOffset
    → user drags a slider to match their TV delay
```

## Key concepts to understand

### The unified SportEvent model
Every sport's API returns data in a different shape. The Producer's job is to normalize all of them into one consistent schema:
```
eventId        — unique ID (UUID generated by producer)
sportType      — enum: SOCCER, FOOTBALL, BASKETBALL, BASEBALL, HOCKEY, FORMULA_1
action         — what happened: "goal", "touchdown", "three-pointer", etc.
participants   — who was involved: Map<String, String> (flexible per sport)
eventTimestamp — epoch milliseconds from the original API response
rawPayload     — the original JSON string, for debugging
```
Every downstream component (Lambda, Bedrock prompts, frontend) works only with this model, never with raw API responses.

### The time-travel buffer
This is pure client-side logic — no server changes needed. The frontend keeps a rolling buffer of the last 500 events. It renders only the ones that are "old enough" based on your offset:
```
visibleEvents = buffer.filter(e => (Date.now() - e.eventTimestamp) / 1000 > offsetSeconds)
```
The slider controls `offsetSeconds`. If your TV is 30 seconds delayed, you set it to 30 and the app shows you exactly what you're about to see on TV.

### Dynamic Bedrock prompts
The Lambda doesn't use one generic prompt. It picks a sport-specific template:
- Soccer: describes a goal/assist/card in soccer terms
- Football: describes a touchdown/interception in NFL terms
- etc.

The `sport_type` field on the event drives this selection.

### IAM least-privilege (important for security)
Each service can only do exactly what it needs — nothing more:
- **Producer**: can only write to Kinesis (not read DynamoDB, not call Bedrock)
- **API service**: can only read from DynamoDB (not write, not touch Kinesis)
- **Lambda**: can write to DynamoDB + call Bedrock (not touch Kinesis directly — ECS event source mapping handles that)
- **Frontend container**: no AWS permissions at all (it only talks to the api service over HTTP)

2. Change the ALB listener from HTTP port 80 to HTTPS port 443

4. Add `domain_name` to `environments/*.tfvars`

No application code changes required.
